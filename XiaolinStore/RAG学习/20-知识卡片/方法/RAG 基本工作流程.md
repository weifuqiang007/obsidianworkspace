# RAG 基本工作流程

## 一句话定义

RAG 通常由离线的知识库构建和在线的查询回答两条流水线组成，两者通过可检索的索引连接。

## 离线：知识库构建

输入是网页、Markdown、PDF 等原始资料，典型步骤是：

1. 收集并解析文档。
2. 清理和标准化文本。
3. 按语义边界切分文档。
4. 为文本块添加标题、来源、时间等元数据。
5. 生成可用于检索的表示，例如向量。
6. 写入检索索引，并保留原文或其定位信息。

## 在线：查询回答

输入是用户问题，典型步骤是：

1. 理解或改写查询。
2. 使用与索引兼容的方式表示查询。
3. 召回候选文本块。
4. 过滤、去重或重排候选结果。
5. 将最有用的文本组织为上下文。
6. 把问题、上下文和回答约束交给 LLM。
7. 返回回答与来源，并记录日志供评估。

本书第二章对两阶段流程的原始描述见 [[RAG应用开发与实战手册-原文#2.2 RAG 的工作流程|来源]]。

## 输入与输出

| 流水线 | 输入 | 输出 |
|---|---|---|
| 知识库构建 | 原始文档 | 文本块、元数据和可检索索引 |
| 查询回答 | 用户问题 | 带来源的自然语言回答 |

## 关键失败点

- 解析丢失表格、标题或图片语义。
- 分块切断关键上下文。
- 查询和文档的表达方式不对齐。
- 召回相关内容后，排序仍把噪声放在前面。
- 上下文过长、矛盾或缺少来源。
- 生成模型没有严格依据上下文回答。

## 复习问题

1. 为什么知识库构建通常放在离线流程？
2. 文档和查询为什么需要兼容的表示方式？
3. 检索到 top-k 以后，为什么还可能需要重排？
4. 如果答案错误，如何判断问题发生在检索还是生成阶段？
