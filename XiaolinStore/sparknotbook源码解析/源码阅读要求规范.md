## 源码阅读规范要求

### 1. 应用初始化分析
需要分析应用启动时加载了哪些内容、实现了哪些功能、初始化了哪些组件。

### 2. 组件模块化拆解
不同的组件（例如：日志、中间件、认证模块等）如果是作者自己实现的，需要分门别类地将实现的方法写在单独的源码解析文档中。然后与源文件建立双向关联。这样关系图谱中可以看到整体架构，未来可以添加更多相关文档，形成完整的知识网络。

### 3. 源码贴图要求

#### 3.3 代码示例
```markdown
#### 1.1 注册流程
- **路径**: `POST /api/auth/register`
- **输入**: UserCreate（用户名、邮箱、密码）
- **处理逻辑**:
  1. 验证用户名唯一性
  2. 验证邮箱唯一性
  3. 密码哈希加密
  4. 创建用户记录
  5. 记录注册日志

**源码实现**:
```python
@router.post("/register", response_model=User)
def register(user: UserCreate, db: Session = Depends(get_db)):
    # 检查用户名是否已存在
    db_user = db.query(UserModel).filter(UserModel.username == user.username).first()
    if db_user:
        raise HTTPException(
            status_code=status.HTTP_400_BAD_REQUEST,
            detail="Username already registered"
        )
    
    # 创建新用户
    hashed_password = get_password_hash(user.password)
    db_user = UserModel(
        username=user.username,
        email=user.email,
        password_hash=hashed_password
    )
    db.add(db_user)
    db.commit()
    db.refresh(db_user)
    
    return db_user
```

### 4. 文档组织结构要求

#### 4.1 业务模块文档组织
每个业务模块（如 auth、note、integrations 等）应建立独立的文档文件夹：

```
sparknotbook源码解析/
├── auth模块/                 # 业务模块文件夹
│   ├── auth模块业务拆解.md    # 主业务文档
│   ├── 依赖分析.md          # 模块间依赖关系
│   └── 源码细节/           # 源码细节文档（可选）
│       ├── 认证流程.md
│       └── 安全机制.md
├── common/                 # 公共组件文件夹
│   ├── utils/              # 工具函数模块
│   │   ├── auth工具.md
│   │   └── 加密工具.md
│   ├── services/           # 服务层模块
│   │   └── user_session.md
│   └── core/              # 核心组件模块
│       ├── config.md
│       └── logger.md
└── 业务模块/              # 其他业务模块...
```

#### 4.2 文档命名规范
- **主业务文档**: `[模块名]模块业务拆解.md`
- **依赖分析**: `[模块名]依赖分析.md`
- **源码细节**: `[功能名]源码分析.md`
- **公共组件**: `[组件名].md`（放在对应子文件夹中）

#### 4.3 链接管理规范
- 使用 Obsidian 的双向链接 `[[文档名]]`
- 每个文档顶部应有"相关文档"部分，列出所有相关链接
- 链接应分类：依赖模块、相关组件、扩展阅读等

#### 4.4 公共方法提取原则
- 如果被多个业务模块使用，提取到 `common/` 文件夹
- 公共工具函数应在使用处添加链接 `[[common/utils/工具名]]`
- 服务层组件应独立成文，建立与使用模块的链接

### 5. 其他要求
- 将其他重复的代码移到专门的模块
- 在使用的地方添加双向链接
- 保持 DRY (Don't Repeat Yourself) 原则
- 确保所有业务功能都有对应的源码实现
- 按照文档组织结构规范整理现有文档
