```python
class ScopePolicy(ABC, Generic[T]):  
    """资源可见性策略抽象。业务模块为每类资源注册一个实现。"""  
  
    @abstractmethod  
    async def build_principal(self, db: AsyncSession, current_user) -> Principal:  
        """从当前登录用户构造 Principal（业务在此查询组织关系并填充领域字段）。"""  
  
    @abstractmethod  
    def visible_filter(self, principal: Principal, model: type[T]) -> ColumnElement:  
        """列表过滤：返回 where 条件（SQL 级，保证列表可扩展）。  
  
        admin 场景应返回永真条件，由调用方拼入主查询。  
        """  
    @abstractmethod  
    def can_access(self, principal: Principal, resource: T) -> bool:  
        """单条访问判定（详情/检索/改删前置），纯 Python，principal 须已含全部所需字段。"""
```
这个类不是「没写完」，它是**故意只写签名、不写实现**的。两个关键词：

- `ABC` = Abstract Base Class（抽象基类）
- `@abstractmethod` = 标记「这个方法是抽象的，只定规矩不定做法」
它是一份**合同/接口**，意思是：「任何想做『资源可见性策略』的类，必须会这三件事——构造主体、列表过滤、单条判定。但具体怎么做，框架不管，你自己写。」

而且抽象类**不能直接实例化**（`ScopePolicy()` 会报错），必须由子类把三个方法都实现一遍，才能 `new` 出对象。谁来实现？就是业务层的 `KbScopePolicy`（[kb_scope_policy.py:80](vscode-webview://154u6p70kifhlkttm7dhe918h50edn35fjhr17lkgc13b7u736l8/ruoyi-fastapi-backend/module_rag/service/kb_scope_policy.py#L80)），它继承 `ScopePolicy` 并把三个方法都写了真正的逻辑。

**类比**：`ScopePolicy` 像「插座标准」（规定有几个孔、多少伏），`KbScopePolicy` 像某个牌子的具体插头。标准不发电，发电的是插头。


###  `_map` 什么时候填充？key/value 是什么？
```python
class ScopeRegistry:  
    """按 resource_key 注册/查找 ScopePolicy。"""  
  
    _map: dict[str, ScopePolicy] = {}  
  
    @classmethod  
    def register(cls, resource_key: str, policy: ScopePolicy) -> None:  
        cls._map[resource_key] = policy  
  
    @classmethod  
    def get(cls, resource_key: str) -> ScopePolicy:  
        if resource_key not in cls._map:  
            raise KeyError(f'未注册资源作用域策略: {resource_key}')  
        return cls._map[resource_key]
```

**`_map` 是「类变量」**：定义在 `class` 体里、不在 `__init__` 里，所以它属于「类本身」，整个程序运行期间只有**一份**，所有地方读写的是同一个字典。这就是为什么你担心「什么时候填充」——它确实需要**提前**被塞进去，后面 `get` 才能取到。

async def school_id_of_dept(db: AsyncSession, dept_id: int | None) -> int | None:  
    """返回 dept_id 所属学校的 dept_id。  
  
    规则：parent_id==0 时自己即学校根；否则取 ancestors 中 0 之后的第一段。

        await db.execute(select(SysDept).where(SysDept.dept_id == dept_id, SysDept.del_flag == '0'))  
    ).scalars().first()  
    if row is None:  
        return None  
    if row.parent_id in (0, None):  
        return dept_id  
    for seg in (row.ancestors or '').split(','):  
        if seg and seg != '0':  
            return int(seg)  
    return None

                school_id=school_id, taught_class_ids=taught_class_ids,  
                owned_student_ids=owned_student_ids, school_dept_ids=school_dept_ids,  
            )  
  
        # 未识别角色：仅能见公共库  
        return KbPrincipal(user_id=user_id, is_admin=False, role=role, role_keys=role_keys)  
  
    # ---- 列表过滤（SQL 级）-------------------------------------  
    def visible_filter(self, p: KbPrincipal, model: type[RagKnowledgeBase]) -> ColumnElement:  
        if p.is_admin:  
            return true()  
  
        conds: list[ColumnElement] = [model.kb_scope == KbScope.PUBLIC]  
  
        # 学校级：同校师生  
        if p.school_id is not None:  
            conds.append(and_(model.kb_scope == KbScope.SCHOOL, model.scope_dept_id == p.school_id))  
  
        # 班级级  
        if p.role == 'student' and p.class_id is not None:  
            conds.append(and_(model.kb_scope == KbScope.CLASS, model.scope_dept_id == p.class_id))  
        if p.role == 'teacher' and p.school_dept_ids:  
            conds.append(and_(model.kb_scope == KbScope.CLASS, model.scope_dept_id.in_(p.school_dept_ids)))  
  
        # 个人级  
        if p.role == 'student':  
            conds.append(and_(model.kb_scope == KbScope.PERSONAL, model.owner_user_id == p.user_id))  
        if p.role == 'teacher' and p.owned_student_ids:  
            conds.append(and_(model.kb_scope == KbScope.PERSONAL, model.owner_user_id.in_(p.owned_student_ids)))  
  
        return or_(*conds) if conds else false()  
  
    # ---- 单条访问判定（纯 Python）------------------------------  
    def can_access(self, p: KbPrincipal, kb: RagKnowledgeBase) -> bool:  
        if p.is_admin:  
            return True  
  
        scope = KbScope(kb.kb_scope) if not isinstance(kb.kb_scope, KbScope) else kb.kb_scope  
  
        if scope == KbScope.PUBLIC:  
            return True  
        if scope == KbScope.SCHOOL:  
            return kb.scope_dept_id == p.school_id  
        if scope == KbScope.CLASS:  
            if p.role == 'student':  
                return kb.scope_dept_id == p.class_id  
            if p.role == 'teacher':  
                return kb.scope_dept_id in (p.school_dept_ids or [])  
            return False  
        if scope == KbScope.PERSONAL:  
            if p.role == 'student':  
                return kb.owner_user_id == p.user_id  
            if p.role == 'teacher':  
                return kb.owner_user_id in (p.owned_student_ids or [])  
            return False  
        return False  
    # ---- 创建时的作用域合法性校验（设计方案 §3.4）---------------

        if p.is_admin:  
            return  # admin 可建任意级别  
        if kb_scope == KbScope.PUBLIC:  
            raise PermissionException(message='仅管理员可创建公共知识库')  
        if p.role == 'teacher':  
            if kb_scope == KbScope.SCHOOL:  
                if scope_dept_id is None or scope_dept_id != p.school_id:  
                    raise PermissionException(message='学校级知识库作用域必须为本人所属学校')  
            elif kb_scope == KbScope.CLASS:  
                if scope_dept_id is None or scope_dept_id not in (p.taught_class_ids or []):  
                    raise PermissionException(message='班级级知识库作用域必须为本人任课班级')  
            else:  
                raise PermissionException(message='教师仅可创建学校级/班级级知识库')  
        elif p.role == 'student':  
            if kb_scope != KbScope.PERSONAL:  
                raise PermissionException(message='学生仅可创建个人知识库')  
        else:  
            raise PermissionException(message='当前角色无权创建知识库')

**key / value 含义**：

||值|含义|
|---|---|---|
|key|`'rag.knowledge_base'`|一个**资源标识字符串**，给这类资源起的名字（模块名.资源名）。控制器和注册处用的是同一个字符串，才能对上号。|

运行时 `_policy()` 返回的就是 `KbScopePolicy` 实例（Q3 讲过），而这个类有 `validate_create` 方法，所以能调。这就是**多态/动态派发**：Python 不看变量声明时是什么类型，运行时对象身上有什么方法就能调什么。

**这里有个值得注意的设计细节**（你敏锐地撞上了）：

- `build_principal / visible_filter / can_access` 是**框架契约**，在抽象基类 `ScopePolicy` 里用 `@abstractmethod` 声明了——意味着框架层「知道」每个策略都一定有这三个方法。
- `validate_create` 是**业务专属**，基类没声明——它只对知识库有意义。

所以严格说，service 层调 `_policy().validate_create(...)` 时，**依赖的是具体业务类型 `KbScopePolicy`，而不是框架抽象**。`_policy()` 的返回类型注解是 `ScopePolicy`（基类），类型检查器其实会提示「基类没有 validate_create」（可能作者用 `# type: ignore` 或没开严格检查绕过了）。在这个场景能跑、也能用，但从「机制/规则分离」的纯粹性看，`validate_create` 这类业务校验放在 service 里直接写、或单独抽个业务接口，会比挂在 policy 上更干净。这是个小瑕疵，不影响功能。

## 把整条线串起来（以你选中的 create 接口为例）

```
请求进来
  ↓ FastAPI 注入
principal = ViewerDependency('rag.knowledge_base')
  → _resolve() 跑
  → ScopeRegistry.get('rag.knowledge_base')  ← 查 _map（启动时已注册）
  → 拿到 KbScopePolicy 实例
  → 调 policy.build_principal(db, user)       ← 造出当前用户身份
  → 返回 KbPrincipal，注入成 principal

controller 第 48 行:
  await KnowledgeBaseService.create(db, data, principal)

service.create 第 23 行:
  _policy().validate_create(principal, ...)   ← 调策略对象的校验方法
    → 不合法就抛 PermissionException，请求结束
    → 合法就继续建库
```

核心三个东西：**`_map` 是登记簿（启动时填）**、**`KbScopePolicy` 实例是干活的策略**、**`KbPrincipal` 是每请求的身份**。框架（resource_scope.py）只提供登记簿和插座标准，业务（kb_scope_policy.py）提供具体插头并自己登记。
